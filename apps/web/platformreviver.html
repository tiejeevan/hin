Data Model

Everything links to users.id. Flexible metric_key counters — no new column per goal.

erDiagram
    users ||--o| user_gamification : has
    users ||--o{ user_stat_counters : tracks
    users ||--o{ user_badges : earns
    users ||--o{ points_ledger : history
    badges ||--o{ badge_rules : defines
    badges ||--o{ user_badges : grants

    users {
        int id PK
        string username
    }

    user_gamification {
        int user_id PK_FK
        int total_points
        int level
    }

    user_stat_counters {
        int user_id FK
        string metric_key
        int value
    }

    badges {
        int id PK
        string name
        string image_url
        bool is_active
    }

    badge_rules {
        int badge_id FK
        string metric_key
        string operator
        int threshold
    }

    user_badges {
        int user_id FK
        int badge_id FK
        datetime earned_at
    }

    points_ledger {
        int user_id FK
        string action_type
        int delta
    }

End-to-End Action Flow

Example: user shares a post. Stats update, points awarded, badges evaluated, level recalculated.

sequenceDiagram
    participant U as User
    participant API as API Route
    participant Hub as Gamification Hub
    participant Stats as Stat Counters
    participant Rules as Rule Engine
    participant DB as Database
    participant N as Notifications

    U->>API: Share post
    API->>API: Insert post_shares
    API->>Hub: processAction(userId, post_shared, postId)
    Hub->>Stats: unique_posts_shared += 1 if new post
    Hub->>Hub: awardPoints(+5)
    Hub->>Rules: evaluate all active badges
    Rules->>DB: counter >= threshold?
    alt threshold met
        Rules->>DB: insert user_badges
        Rules->>N: badge earned notification
    end
    Hub->>Hub: recalculate level
    Hub->>API: pointsEarned, badgesEarned, progress
    API->>U: Response

Important Goals (Master Checklist)

43 point-by-point targets — product, architecture, security, cost, events, and instant delivery.

Product

  1. Keep platform alive — always something to do beyond post and chat.
  2. Admin ON/OFF toggle — when OFF, earning frozen; progress kept.
  3. When ON: points, levels, badges, event wins in real time.
  4. Admin configures badges, points, levels, events dynamically.
  5. Developers own action catalog; admin wires rewards to metrics.
  6. Link everything to users.id only.
  7. Instant rewards — no page refresh.
  8. Rules and internals stay server-side.

Architecture

  1. One hub — processUserAction() only entry point.
  2. Rule engine — generic DB evaluator, not per-badge if statements.
  3. Metric registry — metric_key handlers, not column per badge.
  4. Three layers: points/levels, badges, events/wins.
  5. Feature flag — OFF = zero gamification D1 cost.
  6. Evaluate on write, not read.
  7. Cached counters — increment, don't COUNT(*).
  8. Generic UI components — data-driven.
  9. Passive rules — follower count on followed user.
  10. Idempotent awards — unique (user_id, badge_id).

Security & Client Exposure

  1. Server authoritative — client never awards.
  2. No public rules API — metric keys never to users.
  3. Minimal public DTO — level, badges, goal progress only.
  4. Piggyback g block on normal route responses.
  5. Admin gamification CRUD — admin role only.
  6. Don't rely on hiding Network tab — minimize payload instead.

Cost, Events & Delivery

  1. Free tier: ~100 concurrent OK; 1K accounts with low DAU OK.
  2. Upgrade at ~70K+ requests or D1 writes/day → Paid $5/mo.
  3. R2 badges negligible; plan ledger archival.
  4. Anti-abuse — rate limits, unique counts, daily caps.
  5. Milestone toasts only — not every +1 point.
  6. Time-boxed events — leaderboard, first-to-N, raffle.
  7. Evaluate active events only.
  8. API response updates UI; WebSocket for toasts + bell.
  9. Reuse RealtimeDO — no new realtime infra.

Core Principles

Rules to keep the system future-proof as new goals are added.

  • One identity: Link everything to users.id — not username, not a separate Gamification ID.
  • One hub: processUserAction() is the only entry point. Routes do not contain badge logic.
  • Admin toggle: gamification_enabled — OFF freezes earning with zero D1 cost.
  • Rule engine: Generic evaluator over DB rules — admin creates badges without deploys.
  • Metric registry: New goal = new metric_key handler, not a schema migration per badge.
  • Instant delivery: API response + WebSocket — no page refresh.
  • Minimal client DTO: Rules and metric keys server-only; Network obscurity is not security.
  • Data-driven UI: Generic BadgeGrid and GoalProgress — no React component per badge.
  • Idempotent awards: Unique (user_id, badge_id) — never earn the same badge twice.
  • Passive rules: Follower count updates on the followed user, not the follower.
